Sección 01

Introducción al Problema

Comprender el alcance del reto es fundamental antes de proponer una solución.

¿Qué pide el reto?

Diseñar un componente reutilizable que funcione como motor de reglas para gestionar solicitudes de reembolso médico. Debe controlar estados, validar reglas de negocio y emitir resultados de evaluación.

¿Qué NO es el reto?

  • No es una aplicación completa con interfaz gráfica.
  • No requiere base de datos ni persistencia.
  • No es un sistema monolítico acoplado a un framework.
  • No es un CRUD tradicional.

¿Qué se busca demostrar?

D

Diseño orientado a objetos sólido

Aplica principios SOLID y GRASP en cada decisión arquitectónica.

C

Componente desacoplado

Independiente de interfaz, base de datos y framework externo.

E

Extensibilidad real

Permite agregar nuevos tipos, reglas o estados sin modificar código existente.

Requisitos funcionales clave: Consultar acciones válidas según estado · Ejecutar transiciones controladas · Evaluar solicitudes mediante reglas de negocio · Exponer hallazgos y causas de rechazo.

Reglas de Negocio Obligatorias

R1

Documentación mínima por tipo

Cada tipo de solicitud exige documentos específicos que deben estar adjuntos.

R2

Ventana de tiempo (60 días)

La fecha del evento médico no puede superar los 60 días respecto a la fecha actual.

R3

Monto máximo por tipo

El valor solicitado no debe exceder el tope establecido para cada tipo de solicitud.

R4

Prestador habilitado

El prestador asociado a la solicitud debe existir en un catálogo de prestadores autorizados.

R5

Consistencia básica de datos

Valor > 0, paciente existente, prestador existente, tipo de solicitud definido.

Restricciones del Flujo

  • ✕ No se puede evaluar una solicitud en estado Draft.
  • ✕ No se puede aprobar o rechazar sin evaluación previa.
  • ✕ No se pueden saltar estados intermedios.
  • ✕ Approved y Rejected son terminales: no admiten retroceso.

Sección 06

Motor de Evaluación de Reglas

El EvaluadorSolicitudes recorre secuencialmente cada regla, recopila hallazgos y emite una decisión final.

1

Recibir solicitud

El motor recibe la solicitud a evaluar.

→
2

Obtener política

Se resuelve la política según el tipo de solicitud.

→
3

Ejecutar reglas

Cada ReglaEvaluacion se evalúa y genera un hallazgo.

→
4

Recopilar hallazgos

Todos los HallazgoRegla se agrupan en una lista.

→
5

Emitir decisión

Se determina si es aprobable o rechazable.

¿Qué es un Hallazgo?

Un HallazgoRegla es el resultado individual de evaluar una regla sobre una solicitud. Contiene la identificación de la regla que lo generó, una descripción del resultado y un indicador de si la regla fue cumplida o no.

Ejemplo: La regla ReglaVentanaTiempo evalúa la fecha del evento. Si han pasado 75 días, genera un hallazgo negativo: "La fecha del evento excede la ventana de 60 días permitida."

Decisión Final

✔ Aprobable

Todas las reglas se cumplen. No hay hallazgos negativos. La solicitud puede ser aprobada.

✕ Rechazable

Al menos una regla genera un hallazgo negativo. Los hallazgos se exponen como causas del rechazo.

La decisión se almacena en ResultadoEvaluacion junto con la lista completa de hallazgos, permitiendo transparencia total sobre las causas de aceptación o rechazo.

Sección 07

Aplicación de Principios SOLID

Cada principio SOLID se aplica de forma deliberada en el diseño arquitectónico, asegurando un componente robusto y profesional.

S
Single Responsibility Principle (SRP)

Cada clase tiene una única razón para cambiar. La responsabilidad de evaluar reglas recae exclusivamente en EvaluadorSolicitudes; la de controlar transiciones, en EjecutorAcciones; la de orquestar, en MotorReglasFacade. Ninguna clase acumula responsabilidades ajenas.

Ejemplo concreto: ReglaMontoMaximo solo valida que el monto no exceda el tope. No consulta documentos, no verifica prestadores, no gestiona estados. Si cambia la lógica de montos, solo esta clase se modifica.
O
Open/Closed Principle (OCP)

El sistema está abierto a la extensión y cerrado a la modificación. Para agregar una nueva regla de negocio, basta con crear una nueva clase que implemente ReglaEvaluacion y registrarla. No se modifica ningún código existente.

Ejemplo concreto: Si mañana se requiere validar que el paciente sea mayor de edad, se crea ReglaEdadMinimaPaciente implementando la interfaz ReglaEvaluacion. El EvaluadorSolicitudes la ejecutará sin necesidad de modificar su código interno.
L
Liskov Substitution Principle (LSP)

Cualquier implementación de EstadoSolicitud o ReglaEvaluacion puede sustituir a otra sin alterar el comportamiento esperado del sistema. El EvaluadorSolicitudes no necesita conocer qué regla concreta está ejecutando.

Ejemplo concreto: El EjecutorAcciones trabaja con la interfaz EstadoSolicitud. Ya sea que el estado actual sea DraftState o UnderReviewState, el contrato se respeta: ambos responden a obtenerAccionesValidas() y ejecutarTransicion() de forma coherente con su semántica.
I
Interface Segregation Principle (ISP)

Las interfaces son específicas y cohesivas. ReglaEvaluacion expone solo el método evaluar(). EstadoSolicitud expone solo métodos relacionados con el estado. Los puertos externos (IClock, IPrestadorHabilitadoCatalog) son interfaces mínimas y orientadas al consumidor.

Ejemplo concreto: IClock expone únicamente la capacidad de obtener la fecha actual. No obliga al implementador a proveer zonas horarias, formateo o funcionalidades ajenas. Cada interfaz se diseña según lo que el consumidor necesita, no según lo que el proveedor puede ofrecer.
D
Dependency Inversion Principle (DIP)

Los módulos de alto nivel no dependen de módulos de bajo nivel; ambos dependen de abstracciones. El EvaluadorSolicitudes depende de la interfaz ReglaEvaluacion, no de implementaciones concretas. Los puertos externos (IClock, IPrestadorHabilitadoCatalog) son abstracciones que se inyectan desde fuera.

Ejemplo concreto: ReglaVentanaTiempo recibe una instancia de IClock inyectada. En producción puede ser un reloj real del sistema; en pruebas, un reloj simulado con fecha fija. La regla nunca instancia directamente una dependencia de infraestructura.
Sección 08

Aplicación de Principios GRASP

Los principios GRASP guían la asignación de responsabilidades a las clases del diseño.

HC
High Cohesion (Alta Cohesión)

Cada clase del diseño tiene un conjunto coherente y enfocado de responsabilidades. Las reglas solo evalúan, los estados solo gestionan transiciones, los servicios solo orquestan.

Ejemplo: EvaluadorSolicitudes solo evalúa reglas y construye resultados. No gestiona estados, no controla transiciones, no accede a infraestructura directamente. Su cohesión es máxima.
LC
Low Coupling (Bajo Acoplamiento)

El acoplamiento entre clases se minimiza al depender de interfaces y abstracciones en lugar de implementaciones concretas. Los puertos externos garantizan que el componente no se acopla a infraestructura real.

Ejemplo: ReglaPrestadorHabilitado depende de IPrestadorHabilitadoCatalog (interfaz), no de una base de datos, un servicio REST o un archivo CSV. Cambiar la fuente de datos no afecta a la regla.
PO
Polymorphism (Polimorfismo)

Se usa polimorfismo para evitar condicionales extensos. En lugar de un switch/case sobre el estado actual, cada estado implementa su propio comportamiento. Lo mismo aplica para reglas y políticas.

Ejemplo: En vez de if (estado == "Draft") ... else if (estado == "Submitted") ..., el sistema invoca estado.obtenerAccionesValidas() y cada clase de estado responde según su lógica propia.
CO
Controller (Controlador)

MotorReglasFacade actúa como controlador del caso de uso. Recibe las solicitudes externas, delega el trabajo a los servicios especializados y devuelve el resultado. No contiene lógica de negocio directa.

Ejemplo: Cuando un cliente invoca motorReglas.evaluarSolicitud(solicitud), la fachada delega a EvaluadorSolicitudes. Si invoca ejecutarAccion(solicitud, accion), delega a EjecutorAcciones. La fachada coordina, no ejecuta.
CR
Creator (Creador)

Cada clase crea los objetos que necesita y que naturalmente le pertenecen. El EvaluadorSolicitudes crea los objetos ResultadoEvaluacion y HallazgoRegla como parte de su proceso de evaluación.

Ejemplo: Es el EvaluadorSolicitudes quien tiene los datos y el contexto para construir un ResultadoEvaluacion con su lista de hallazgos. Ninguna otra clase tiene la información necesaria para crearlo correctamente.
Sección 09

Aplicación de DRY, KISS y YAGNI

Estos tres principios complementarios guían las decisiones de diseño hacia la simplicidad y la eficiencia.

DRY
Don't Repeat Yourself

Toda lógica tiene un único punto de definición. No se duplican reglas, políticas ni validaciones de estado.

Ejemplo 1: Los documentos requeridos y montos máximos se definen una sola vez en cada PoliticaTipoSolicitud. Las reglas que los necesitan consultan la política en lugar de redefinir esos datos internamente.
Ejemplo 2: La lógica de "qué acciones son válidas" vive exclusivamente en cada implementación de EstadoSolicitud. Ningún servicio duplica esa información.
KISS
Keep It Simple, Stupid

El diseño es directo y comprensible. No se introducen patrones por moda ni abstracciones innecesarias. Cada patrón utilizado resuelve un problema concreto.

Ejemplo 1: El patrón State se usa porque hay transiciones reales entre estados con comportamiento diferente, no como ejercicio académico. Sin este patrón, habría condicionales extensos y frágiles.
Ejemplo 2: La fachada MotorReglasFacade expone solo tres métodos claros: consultarAcciones, ejecutarAccion y evaluarSolicitud. La interfaz pública es mínima y comprensible.
YAGNI
You Aren't Gonna Need It

Solo se implementa lo que el reto exige. No se anticipan funcionalidades hipotéticas como historial de cambios, notificaciones o reportes. La extensibilidad está garantizada por el diseño, no por código preventivo.

Ejemplo 1: No se implementa un sistema de auditoría ni de eventos. Si en el futuro se necesita, las interfaces existentes permiten añadirlo sin refactorizar el núcleo.
Ejemplo 2: No se crean repositorios, DTOs de transferencia ni capas de serialización. El componente es un motor de reglas puro: recibe objetos de dominio, los evalúa y devuelve resultados.
Síntesis: DRY evita la duplicación; KISS asegura que el diseño sea comprensible; YAGNI impide la sobreingeniería. Juntos, producen un componente enfocado, limpio y fácil de mantener.
Sección 10

Decisiones Arquitectónicas Clave

Cada decisión de diseño responde a un razonamiento deliberado. A continuación se justifican las más relevantes.

¿Por qué una sola entidad SolicitudReembolso?

SolicitudReembolso es la entidad raíz del dominio: representa el concepto central alrededor del cual giran todas las operaciones. Centralizar su definición evita redundancia y asegura un modelo conceptual claro y cohesivo. Todas las reglas, estados y servicios operan sobre esta entidad, lo cual simplifica el diseño.

¿Por qué políticas por tipo de solicitud?

Los requisitos de documentación y montos máximos varían según el tipo de solicitud. Encapsular estas variaciones en políticas (MedicamentoPolicy, UrgenciasPolicy, etc.) permite que las reglas de evaluación sean genéricas y reutilizables. Agregar un nuevo tipo de solicitud solo requiere crear una nueva política, sin tocar reglas existentes.

¿Por qué estados polimórficos?

El patrón State elimina la necesidad de cadenas de condicionales (if/switch) en cada operación que dependa del estado. Cada estado encapsula sus transiciones válidas, lo que garantiza que las restricciones del flujo se respeten localmente. Además, facilita agregar nuevos estados sin modificar los existentes.

¿Por qué interfaces para reglas y puertos?

Las interfaces permiten inversión de dependencias y extensibilidad. Las reglas implementan ReglaEvaluacion, lo que permite añadir nuevas reglas sin modificar el evaluador. Los puertos (IClock, IPrestadorHabilitadoCatalog) desacoplan el componente de la infraestructura, facilitando las pruebas unitarias con mocks y stubs.

¿Por qué no acoplar a base de datos ni UI?

El reto pide un componente reutilizable, no una aplicación. Acoplar a una base de datos o interfaz gráfica violaría el principio de responsabilidad única del componente y lo haría dependiente de tecnologías que no le competen. Los puertos externos permiten que el consumidor suministre la infraestructura según sus necesidades.

¿Por qué una fachada como punto de entrada?

MotorReglasFacade simplifica el uso del componente al exponer una API mínima y clara. El consumidor no necesita conocer la estructura interna (evaluador, ejecutor, reglas, estados). La fachada actúa como puerta de entrada que orquesta la colaboración entre los servicios internos de forma transparente.

Beneficios de la Solución

Extensibilidad

🔌

Resultado: Un componente que se puede probar, extender, reutilizar y mantener con confianza, alineado con las mejores prácticas de ingeniería de software.

El resultado es un componente que no depende de infraestructura, se puede probar unitariamente y se extiende sin modificar código existente.
